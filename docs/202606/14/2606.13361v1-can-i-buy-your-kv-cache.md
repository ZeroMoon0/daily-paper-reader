## 摘要
如今，全球各地的AI代理都在重复着同样荒谬的行为：要阅读一份文档，它们各自从零开始重新计算。每个代理都会重新运行预填充——这是大模型计算最密集的步骤——在相同的文本上，却只为了重建一个与前一个代理刚刚构建的完全相同的键值（KV）缓存。相同的答案被计算了百万次。我们提出一个几乎简单到冒犯的提议：只计算一次。让发布者预计算文档的KV缓存，然后让其他每个代理购买加载它的权限，从而跳过预填充。这是可行的，而且是令牌精确的：加载预计算的KV并继续，与从头开始预填充完全匹配（24/24个贪婪令牌，且在logits层面），且无精度损失。在Qwen3-4B上，重用比预填充在计算上便宜9-50倍，且差距随长度扩大（预填充的注意力随L^2缩放），因此单次重用即可收回成本。然后关键问题在于：KV存储在何处。传输KV是失败的，因为KV几乎不可压缩，因此每次加载的出口成本比其节省的预填充成本更高。将其托管在提供方侧，正如生产环境中的提示缓存那样，完全消除了出口成本。收益规模由我们测量的计算节省决定：向8000万代理提供一份热门的3774令牌文档，重新预填充的成本约为150万美元，而重用的计算成本仅为约3万美元（节省49.7倍）。API收取的0.1倍缓存读取关税在满足这一测量上限的同时，向用户提供10倍的折扣，因此10倍是一个下限，而测量的约50倍计算节省超出了它，与物理极限约50倍的差距则是提供方的利润：每份热门文档数百万美元。我们构建了由此产生的面向代理的原生预填充CDN，并将无损KV压缩和跨支付方结算层留作开放问题。

## Abstract
Right now, across the world, AI agents are repeating the same absurd act: to read one document, they each recompute it from scratch. Every agent re-runs prefill, the most compute-intensive step a large model takes, over identical text, only to rebuild a key-value (KV) cache identical to the one the agent before it just built. The same answer, computed a million times. We make a proposal that is almost offensively simple: compute it once. Let a publisher precompute a document's KV cache, and let every other agent buy the right to load it and skip prefill. It works, and it is token-exact: loading a precomputed KV and continuing matches prefilling from scratch (24/24 greedy tokens, and at the logits level), with no accuracy cost. On Qwen3-4B, reuse is 9-50x cheaper in compute than prefill, and the gap widens with length (prefill's attention scales with L^2), so a single reuse already pays it back. Then the part that matters: where the KV lives. Shipping it fails, because KV is nearly incompressible, so per-load egress costs more than the prefill it saves. Hosting it provider-side, exactly as production prompt-caching works, removes egress entirely. The size of the prize is set by our measured compute saving: serving one hot 3774-token document to 80M agents costs ~$1.5M to re-prefill but only ~$0.03M of reuse compute (49.7x less). The 0.1x cache-read tariff APIs charge passes a 10x discount to users while sitting inside this measured envelope, so the 10x is a floor that the measured ~50x compute saving clears, and the gap to the physical ~50x is provider margin: millions of dollars per popular document. We frame the resulting agent-native prefill CDN and leave lossless KV compression and a cross-party payment layer as the open problems.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：当前AI代理（agent）在读取同一份文档时，每个代理都会独立地从头执行最耗计算的“预填充（prefill）”步骤，重复构建相同的键值（KV）缓存，造成巨大的计算浪费。例如，一份热门文档被8000万代理读取，就会重复执行8000万次预填充。
- **整体含义**：论文提出一种极端简单的方案——让发布者预先计算文档的KV缓存并使其成为一个可购买的**持久化制品**，其他代理支付一定费用加载该缓存，从而完全跳过预填充。这相当于构建一个“预填充CDN”，将计算缓存而非内容缓存。该方案不仅能节省大量计算，还揭示了在这种经济模式下，托管侧缓存（provider-side）比传输KV缓存更优，因为KV几乎不可压缩，传输成本反而高于节省的计算成本。论文通过实验验证了可行性，并指出了开放问题（无损KV压缩、跨方支付层）。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：将文档的KV缓存视作一个可复制、可销售的计算制品。发布者计算一次，存储为制品；代理购买后加载制品，然后从缓存继续生成输出。
- **关键技术细节**：
  - **精确前缀重用**：论文仅研究文档作为共享前缀的简单情况（后续追加）。因为因果注意力机制，前缀的KV不依赖于后续token，因此加载预计算KV完全等价于从头预填充（token-exact）。
  - **算法描述**：
    - **保存KV（发布者）**：运行一次前向传播得到所有层的KV张量，序列化到磁盘，附带元数据（模型ID、精度、长度）。
    - **从KV生成（代理）**：反序列化制品，确保模型与精度匹配；设置正确的position ids（从L开始）和全注意力掩码；然后只计算查询（query）的KV并解码输出token。
  - **成本模型**：
    - 从头计算总成本 = N × C_prefill
    - KV重用总成本 = C_prefill + N × C_load
    - 盈亏平衡点 N* = C_prefill / (C_prefill - C_load)，当 C_load << C_prefill 时，N* ≈ 1，即第二次阅读就开始节省。
    - 进一步考虑存储和传输成本后，传输成本可能抵消收益，因此主张**托管在提供方数据中心**（类似于生产中的prompt caching），消除出口费用。
- **关键公式**：论文给出两个算法伪代码（Algorithm 1: SAVE KV, Algorithm 2: GENERATE FROM KV），以及成本公式。

## 6. 论文的主要结论与发现

- **正确性**：加载预计算KV并续写，与从头预填充在贪婪解码下完全令牌精确（24/24匹配，logits差值≤0.02）。因此无准确率损失。
- **计算节省巨大**：在同一设备上，重用一次续写的计算成本仅为完整预填充的1/8.6到1/49.7，且随上下文长度增加而增加（注意力O(L²) vs 续写接近常数）。具体：255 token节省8.6倍，3774 token节省49.7倍。
- **盈亏平衡几乎立即**：N* ≈ 1.02~1.13，即第二次使用就开始净节省。
- **传输KV不可行**：以3774 token为例，完整KV制品（fp16）约557 MB，按0.09$/GB出口费计算，单次传输成本约0.049美元，远高于节省的0.019美元预填充成本（按输入token定价）。因此必须采用托管侧缓存。
- **托管侧缓存的经济潜力**：托管时省去出口费，物理计算成本仅为预填充的1/49.7。若API收取0.1倍缓存读取费（即用户折扣10倍），则提供方仍有约5倍利润率（49.7/10 ≈ 5）。按服务8000万代理计算：重新预填充需约150万美元，而托管复用仅需约3万美元，节省49.7倍。
- **量化会破坏精确性**：int8量化使制品大小减半（102→51 MB），但token精确度从100%下降到50%（16/32），输出虽仍流畅但有差异。因此无损压缩或选择性精度方案是必要的。
