## 摘要
现代大型语言模型（LLM）服务集群将推理请求分布到不同GPU上的多个工作进程中，但在大规模部署中故障普遍存在。当某个工作进程发生故障时，集群会同时丢失该故障进程的GPU驻留键值（KV）缓存和服务容量，导致幸存的工作进程在从头重新运行中断请求的同时，吸收重定向的流量。现有的容错系统要么从头重启中断的请求，要么从存储在固定相邻工作进程上的检查点恢复KV缓存，但这两种方法在恢复工作时未考虑当前集群负载，并且在模型重新加载期间使恢复的工作进程处于空闲状态。我们提出LUMEN，一种容错LLM服务系统，将恢复视为一个负载感知的协调问题，涉及三个决策点：故障前的检查点放置、故障时的中断请求分发以及模型重新加载期间的服务容量恢复。我们通过原型实验和大规模模拟对LUMEN进行评估，展示了在服务时间和恢复时间上的显著改进。

## Abstract
Modern large language model (LLM) serving clusters distribute inference requests across multiple worker processes on different GPUs, but failures are prevalent at scale. When a worker fails, the cluster simultaneously loses the failed worker's GPU-resident key-value (KV) caches and serving capacity, leaving surviving workers to absorb the redirected traffic while re-running interrupted requests from scratch. Existing fault-tolerant systems either restart interrupted requests from scratch or restore KV caches from checkpoints stored on a fixed neighboring worker, but both approaches route recovery work without considering current cluster load and leave the recovering worker idle during model reload. We present LUMEN, a fault-tolerant LLM serving system that treats recovery as a load-aware coordination problem across three decision points: checkpoint placement before failures, interrupted-request distribution at failure time, and serving capacity restoration during model reload. We evaluate LUMEN using both prototype experiments and large-scale simulations and demonstrate significant improvements in serving and recovery times.

### 1. 论文的核心问题与整体含义 (研究动机和背景)

- **核心问题**：在大型、分布式的LLM（大语言模型）服务集群中，GPU或节点故障时有发生。故障会导致：
    - **KV缓存丢失**：故障工作进程（Worker）GPU上的Key-Value缓存丢失，中断的请求必须从头开始重新计算，耗时巨大。
    - **服务容量损失**：故障工作进程的算力损失，导致幸存工作进程需要吸收重定向的流量，造成队列拥堵和延迟飙升。
- **现有方法的不足**：
    - **Stop-and-Restart**：所有中断请求在幸存工作进程上从头开始重算，恢复时间长，且给幸存节点带来巨大负载。
    - **Fixed-Checkpointing (例如DéjàVu)**：将KV缓存备份到固定的邻居工作进程，但未考虑该邻居的负载状况，容易形成恢复瓶颈。
    - **共同缺点**：两种方法在模型重载期间都让恢复中的工作进程闲置，浪费了宝贵的恢复时间。
- **整体含义**：论文旨在设计一种**负载感知**的协调恢复机制，以有效管理故障恢复过程中的三个关键决策点（检查点放置、中断请求分发、容量恢复），从而显著缩短服务中断时间，提升系统弹性。

### 2. 论文提出的方法论 (LUMEN系统)

LUMEN将故障恢复视为一个**负载感知的协调问题**，主要由以下三个协同机制构成：

- **机制一：负载感知的KV检查点 (`Load-aware KV Checkpointing`)**
    - **核心思想**：在为每个请求选择KV缓存的检查点（备份）持有者（Checkpoint Holder）时，**动态评估所有幸存工作进程的当前负载**。
    - **关键技术**：
        - 引入一个中心化控制器，维护一个包含队列延迟(`q_w`)和预期恢复压力(`p_w(r)`)的负载表。
        - 通过公式 `h(r) = arg min_{w in F(r)} (q_w + λ * p_w(r))` 来选择检查点持有者。其中 `F(r)` 是拥有足够存储空间的候选工作进程集合。`λ` 是可调权重。
        - **增量式检查点流水线**：KV页面（KV pages）被异步、增量地通过Peer-to-Peer方式流式传输到检查点持有者的CPU内存中，完全解耦于GPU推理的临界路径，不增加正常服务时的延迟。
        - 每个页面携带一个**页面标签**（由token序列的哈希值生成），用于在恢复时快速定位和恢复最长连续KV前缀。

- **机制二：局部感知的恢复调度 (`Locality-aware Recovery Scheduling`)**
    - **核心思想**：在故障发生时，智能地分发中断请求，**在最大化KV缓存复用和避免负载热点之间取得平衡**。
    - **关键技术流程**：
        1. **初始分发**：中断的请求首先被路由到其指定的检查点持有者，以复用KV缓存。
        2. **负载再平衡**：如果某个检查点持有者的总负载（包括新分配的中断请求）超过集群平均水平，则被视为过载。
        3. **贪婪迁移**：将过载检查点持有者上的请求（按**已检查点的KV大小升序排列**）迁移到最空闲的工作进程。优先迁移**短前缀**的请求，以最小化重计算带来的损失。

- **机制三：投机辅助的渐进恢复 (`Speculation-assisted Progressive Recovery`)**
    - **核心思想**：在恢复中的工作进程加载完整目标模型权重的**后台期间**，使其立即利用**投机解码**技术提供临时服务，而不是闲置等待。
    - **关键技术流程 (分为四个状态)**：
        1. **LOADING_DRAFT**：加载一个轻量级的草稿模型（Draft Model）。
        2. **ASSIST**：与负载最高的一个幸存工作进程配对。草稿模型快速生成草稿token，由配对幸存工作进程的**目标模型**在一个**融合的批量前向传播**中进行验证。所有操作是异步的，幸存工作进程不会因此被阻塞。
        3. **HOTSWAP**：后台完成目标模型权重加载（到主机内存）后，快速将其传输到GPU，并切换掉草稿模型。
        4. **FULL_SERVICE**：恢复工作者以完整服务能力正常运行。

### 7. 优点

- **创新的负载感知视角**：将故障恢复从静态的“备份-恢复”模式转变为动态的、负载感知的协调调度问题，这是核心创新点。
- **系统设计巧妙**：
    - 增量异步检查点流水线设计，几乎没有稳态开销。
    - 将投机解码这一加速技术创造性地用于故障恢复，解决了恢复中机器的闲置问题。
    - 通过贪婪迁移算法，在复用KV缓存和负载均衡之间取得了很好的平衡。
- **实验验证全面**：通过原型和仿真两种方式，在多个维度上验证了方法的有效性，结果令人信服。

### 8. 不足与局限

- **实验规模有限**：原型实验最大规模为8块GPU。虽然仿真实验扩展到了64个worker，但真实的大规模生产环境（数千甚至上万GPU）下的表现仍有待验证。
- **依赖中心化组件**：LUMEN依赖一个中心化的控制器和网关。虽然论文提及可以通过“标准复制”实现高可用，但未深入探讨中心化组件本身的故障恢复、单点瓶颈和复杂度问题。
- **未覆盖所有故障场景**：论文假设一个请求只有一个KV缓存检查点副本。如果检查点持有者本身发生故障，请求将退化为Full Recompute。多副本（Multi-replication）被列为未来工作。
- **网络假设**：原型实验在10Gbps网络上进行，而现代GPU集群可能使用更高带宽的InfiniBand网络。在网络性能更好的情况下，LUMEN的负载感知调度和渐进恢复的收益可能会有变化，但论文未深入讨论其敏感性。
