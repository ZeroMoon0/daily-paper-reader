## 摘要
大语言模型训练任务的规模要求在大型GPU集群上进行并行化规划。由于随着时间的推移添加了不同类型的GPU和互连，这些GPU集群变得越来越异构。自动大语言模型并行化工具可以搜索并行化方案，但在异构GPU环境中面临搜索空间爆炸的问题。为了在异构GPU集群中使搜索可行，并行化工具通常会省略某些并行化类型（例如专家并行）或内存节省技术（例如ZeRO），这导致方案质量下降。我们描述了Tangram，一个通过将并行化规划与GPU异构性解耦，从而在异构GPU集群中利用现有不考虑异构性的LLM并行化工具的系统。为此，Tangram利用了两个洞察：（1）由于批量采购导致一组具有相似计算、内存和连接性的GPU，Tangram可以将这样的同构GPU岛暴露给现有的并行化工具；（2）并行化工具通常先对模型进行分区，然后对分区进行并行化。Tangram可以将分配给GPU岛的这些模型切片组合成工作负载均衡的流水线，以实现高吞吐量。Tangram通过一个狭窄的API与现有的并行化工具集成，该API依赖于模型切片/岛对的枚举。与当前的异构并行化工具（Metis和Sailor）相比，Tangram实现了高达2.3倍的训练吞吐量提升，并通过修剪枚举的计划扩展到大型GPU集群。

## Abstract
The scale of LLM training jobs requires parallelization planning over large GPU clusters. Due to different GPU types and interconnects added over time, these GPU clusters are increasingly heterogeneous. Automatic LLM parallelizers can search for parallelization plans but face an exploding search space with heterogeneous GPUs. To make search tractable in heterogeneous GPU clusters, parallelizers often omit types of parallelism (e.g., expert parallelism) or memory-saving techniques (e.g., ZeRO), which results in worse plans.   We describe Tangram, a system that enables the use of existing heterogeneity-unaware LLM parallelizers in heterogeneous GPU clusters by decoupling parallelization planning from GPU heterogeneity. For this, Tangram exploits two insights: (1) since bulk purchases result in sets of GPUs with similar compute, memory, and connectivity, Tangram can expose such homogeneous GPU islands to existing parallelizers; and (2) parallelizers commonly first partition models and then parallelize partitions. Tangram can compose such model slices, assigned to GPU islands, into work-balanced pipelines for high throughput. Tangram integrates with existing parallelizers through a narrow API, which relies on the enumeration of model-slice/island pairs. Tangram achieves up to 2.3x higher training throughput than current heterogeneous parallelizers (Metis and Sailor) and scales to large GPU clusters by pruning enumerated plans.

- **整体含义**：作者提出 **Tangram**，一种通过将 GPU 异构性“隐藏”起来，让现有的“同构假设”并行化工具（如 Alpa、Aceso）也能高效工作于异构集群的系统。其核心思路是：把异构集群划分成若干内部同构的 GPU 岛，再通过模型分片和流水线组合，将同构并行化工具产出的局部方案拼成全局异构方案。

## 2. 论文提出的方法论：核心思想、关键技术细节

### 核心思想
- **GPU 岛抽象（§4.1）**：利用批量采购带来的硬件相似性，将集群中计算、内存、节点内互联带宽相似的 GPU 节点聚合成“同构 GPU 岛”。岛内满足（R1）兼容现有并行化工具的输入形式；（R2）内部充分同构；（R3）岛间通信带宽不会成为严重瓶颈。通过可容忍的阈值（compute、memory、intra-node interconnect 的容忍度）进行贪心合并，保持岛数量少（通常 2-6 个）。
- **模型分解（§4.2）**：将整个 LLM 切分为连续的“模型切片”（model slice）。利用并行化工具已有的 `slice_model()`、`concatenate()` 等接口，将模型原子（如一层 Transformer）组合成不同大小的切片。
- **枚举与剪枝（§5.1）**：枚举所有可能的模型切片/GPU 岛对，并行调用同构并行化工具生成局部并行方案。通过三类剪枝策略大幅减少搜索：
  - **冗余剪枝**：相同组成的切片只调用一次。
  - **不平衡剪枝**：丢弃计算需求与岛容量明显不匹配的配对。
  - **不可行剪枝**：丢弃因 GPU 内存不足而无法并行化的配对。
- **流水线组合（§5.2）**：将局部方案通过流水线并行（PP）组合成全局方案。使用 1F1B 调度，以尽量减少激活内存占用。通过动态规划算法，在允许的阶段不平衡约束下，找出工作负载最均衡的流水线序列，以瓶颈阶段的最小延迟为目标进行优化。

### 关键技术细节（算法流程简述）
1. **岛构建**：获取各 GPU 节点的计算、内存、节点内带宽、节点间带宽指标 → 按计算/内存容忍度分组 → 按节点内带宽容忍度分离 → 用凝聚层次聚类合并高带宽节点 → 输出不相交的 GPU 岛。
2. **模型切片**：调用并行化工具的 `slice_model()` 获得模型原子（如每层）→ 通过 `concatenate()` 组合成不同层数的切片。
3. **枚举与并行查询**：生成所有可能的 (切片, 岛) 对 → 应用剪枝政策 → 并行发送到多个工作节点执行 `parallelize()`。
4. **流水线搜索**：对每个可行方案估算阶段延迟 → 按最小瓶颈贪心构造流水线序列 → 动态规划记忆化搜索 → 允许逐步放宽不平衡约束直到无法改进 → 输出全局并行方案。

## 3. 实验设计：数据集 / 场景、benchmark、对比方法

## 6. 论文的主要结论与发现

1. Tangram 在异构 GPU 集群上相比当前最先进的异构并行化器 Metis 和 Sailor，训练吞吐量提升最高达 **2.3 倍**（图 8 中 12xT4+4xV100 场景）。
2. 在与同构并行化器（Aceso）对比时，Tangram 在异构集群上仍能实现 **1.4× 平均加速**，因为 Aceso 可以支持 activation recomputation 等内存优化而 Metis/Sailor 不支持。
3. 在 MoE 模型上，Tangram 利用 Alpa 的 EP 支持获得 1.1×-2.1× 加速；对现代 MoE（Mixtral/Qwen3），结合 Galvatron 的 FSEP 也能获得 1.1×-1.5× 加速。
4. 在纯同构集群上，Tangram 完全匹配底层同构并行化器的性能，验证了无退化。
5. 剪枝政策将规划时间从 5.1 小时缩减至约 12 分钟（27 倍加速），且不损失方案质量。
6. 随着 GPU 岛数增加（1→4），规划时间从 31 秒增长到约 16 分钟，仍可接受。

## 7. 优点

- **创新性解耦**：通过 GPU 岛抽象和模型切片流水线，首次实现了“异构性隐藏”，使得功能更丰富的同构并行化器可以直接用于异构集群。
- **通用接口**：提供窄 API，可轻松集成 Alpa、Aceso 等多种并行化器，甚至支持 Galvatron 的 MoE 运行时。
- **高效的剪枝策略**：冗余、不平衡、不可行三类剪枝配合并行枚举，大幅降低规划开销。
- **流水线组合优化**：动态规划 + 瓶颈松弛，能找到工作负载均衡的全局方案。
- **实验扎实**：覆盖多个异构程度、多种模型架构，对比了主流并行化器，并做了消融和扩展性分析。

## 8. 不足与局限

- **未在超大规模集群验证**：实验最大节点数为 32 GPU，岛数最多 4 个。真实生产环境可能包含数百至数千 GPU，岛数更多，规划时间的增长（O(岛数×模型层数)）可能成为瓶颈。
- **依赖并行化器本身的限制**：Tangram 能提升的方案质量受限于所集成的同构并行化器的功能。例如，若后者不支持 activation offloading，Tangram 也无法利用该技术。
- **模型分片粒度固定为层**：如果模型层间计算/内存差异大（如某些 MoE 层与密集层混合），以层为原子可能导致岛间负载不均。论文未探讨更细粒度（如算子级别）的切分。
- **流水线组合的局限性**：仅使用 1F1B 调度，未尝试更先进的调度（如 interleaving 1F1B）可能影响性能。
- **实验未覆盖长时间训练的实际扰动**：如网络拥塞、GPU 故障等动态因素，T angram 的静态规划可能无法适应。
- **缺乏对成本/能耗的衡量**：论文仅关注训练吞吐量，未讨论异构集群使用方式的能效比或经济成本。
