## 摘要
多轮对话是LLM应用中的基本场景，广泛应用于聊天机器人和AI代理。随着对话的推进，历史token不断累积。现有系统通过缓存其键值对（KV）来避免重复计算。然而，有限的GPU内存（HBM）容量常常迫使这些KV缓存被卸载到CPU内存或SSD，导致随着上下文增长，KV缓存重载的延迟代价越来越高。同时，HBM容量受限也限制了最大推理长度，从而限制了对话中可支持的轮数。为了解决这两个挑战，我们提出了SwiftCache，一种协同推理系统，使异构模型能够在服务器内共享未充分利用的GPU内存和NVLink带宽。具体来说，KV缓存需求低的模型将其空闲GPU内存捐赠给高需求模型存储前缀缓存，从而允许跨模型通过NVLink共享KV缓存，避免缓慢的PCIe传输。SwiftCache还通过仅在本地GPU内存中保留当前活跃层的KV缓存来进一步减少内存压力，从而支持更长上下文的推理。我们在实际工作负载上的实验表明，与vLLM和SGLang相比，SwiftCache将P99首token延迟（TTFT）最多降低69%，并将最大上下文长度最多扩展3.98倍，同时对同驻模型的干扰极小。

## Abstract
Multi-turn conversation is a fundamental scenario in LLM applications, widely used in chatbots and AI agents. As the conversation evolves, historical tokens accumulate continuously. Existing systems cache their key-value (KV) pairs to avoid redundant computation. However, limited GPU memory (HBM) capacity often forces these KV caches to be offloaded to CPU memory or SSD, making KV cache reloads increasingly costly in terms of latency as the context grows. Meanwhile, the constrained HBM capacity also limits the maximum inference length, thereby restricting the number of turns that can be supported in a conversation.   To address these two challenges, we propose SwiftCache, a collaborative inference system that enables heterogeneous models to share underutilized GPU memory and NVLink bandwidth within a server. Specifically, models with low KV cache demand donate idle GPU memory to store the prefix cache of high-demand models, allowing cross-model KV cache sharing over NVLink and avoiding slow PCIe transfers. SwiftCache further reduces memory pressure by keeping only the KV cache of the currently active layer in local GPU memory, thereby enabling longer-context inference. Our experiments on real-world workloads show that SwiftCache reduces P99 time-to-first-token (TTFT) by up to 69% and extends maximum context length by up to 3.98x compared to vLLM and SGLang, with minimal interference to co-located models.

### 论文深度解析：SwiftCache — 面向多轮对话的高效LLM服务，支持异构KV缓存共享

#### 1. 核心问题与研究动机

-   **核心问题**：在LLM的多轮对话场景中，随着对话轮次增加，历史tokens不断累积，导致两个关键挑战：
    - **KV缓存传输瓶颈**：有限的GPU显存（HBM）迫使历史KV缓存被卸载到速度较慢的CPU内存或SSD。当新一轮对话开始时，从这些慢速存储介质通过PCIe总线将历史KV缓存重新加载回GPU，成为主要的延迟瓶颈，尤其是P99的首Token延迟（TTFT）会显著增加。
    - **HBM容量限制推理长度**：GPU显存容量限制了单个请求的最大上下文长度。尽管模型声称支持超长上下文（如1M tokens），但实际部署中因HBM不足，所能支持的最大长度远小于理论值，这限制了多轮对话的可持续性。
-   **研究动机**：现有系统（如vLLM、SGLang）的层次化缓存策略（GPU->CPU->SSD）无法有效解决上述问题。作者观察到两个未被充分利用的资源：（1）同一服务器上不同模型间的空闲GPU显存；（2）不同模型GPU之间的高速NVLink带宽。因此，他们旨在通过跨模型协作来优化内存利用和通信效率。

#### 2. 方法论：核心思想与技术细节

-   **核心思想**：SwiftCache是一种多模型协同推理系统。它让一个高KV缓存需求的模型（Master）与一个或多个低需求的模型（Worker）共享同一服务器。Master利用Worker的空闲GPU显存和NVLink高速通道来存储和传输其KV缓存，从而避免慢速的PCIe传输，并大幅扩展推理长度。
-   **关键技术细节**：
    -   **Master缓存管理器**：
        -   **层级流式缓存**：这是扩展推理长度的关键。在推理过程中，Master的本地GPU仅存储当前正在执行的Transformer层的KV缓存。其他所有层的KV缓存都被流式存储在Workers的GPU上。需要时，通过NVLink预先加载到本地的层级流式缓存中。该机制将单层缓存的空间释放给整个推理过程，使得Master的本地显存能容纳更多token。
        -   **常规缓存**：如果Master的本地显存还有剩余，这部分空间用于存储一些最常访问的KV缓存，以减少跨GPU加载的次数，实现最低延迟访问。
        -   **KV传输与计算重叠**：KV缓存的加载和存储操作与模型的FFN层计算等任务通过NVLink进行异步重叠，几乎完全隐藏了跨GPU通信的开销。
    -   **Worker缓存管理器**：
        -   **弹性缓存**：Worker不能永久将全部KV缓存空间借给Master，因为它需要服务自身的工作负载。SwiftCache提出了“弹性缓存”机制，支持Worker在O(1)时间内动态收缩或扩展分配给Master的缓存。
        -   **块主导布局**：传统系统采用“层主导”的KV缓存布局（`[n_layers, n_blocks, elems]`），调整缓存大小需要移动大量数据（O(n_layers * n_blocks)）。SwiftCache创新性地提出“块主导”布局（`[n_blocks, n_layers, elems]`）。在此布局下，所有层中属于同一个block的数据在内存中连续存储，调整大小（如新增或移除一个block）只需在内存末尾操作，时间复杂度降为O(1)。
    -   **Master-Worker协调**：通过协调器模块，Worker根据自身工作负载（ScaleUp/ScaleDown函数）动态决定借出或回收缓存容量。为了保证内存对齐，借出/回收的最小单位是Master和Worker块大小的最小公倍数（Least Common Multiple, LCM）所定义的“最小弹性单元”。

-   **公式（文字说明）**：
    -   **最小弹性单元（MEU）计算**：首先计算Master和Worker模型的单个KV块元素数（`block_elems`），然后求二者的最小公倍数（`LCM_elem`）。最后，`Master的MEU = LCM_elem / Master的block_elems`，`Worker的MEU = LCM_elem / Worker的block_elems`。任何借出或回收操作都必须是MEU的整数倍。
    -   **最大上下文长度**：Master的最大上下文长度 = （层级流式缓存容量 + 常规缓存容量）× 块大小。层级流式缓存容量受限于所有Worker能提供的总缓存空间和Master本地显存大小中的较小值。

#### 7. 方法优缺点总结

-   **优点（亮点）**：
    -   **新颖的跨模型协作范式**：首次系统性地提出利用服务中不同模型（异构模型）的空闲资源（GPU显存和NVLink带宽），解决了单一模型推理的资源瓶颈，是思想上的创新。
    -   **高效的内存管理**：块主导的弹性缓存布局是工程上的精巧设计，使得缓存大小调整（借出/回收）的时间复杂度从线性降低到常数，大大提高了系统的灵活性和可操作性。
    -   **兼顾性能与公平**：通过层级流式缓存和弹性缓存，既大幅提升了Master模型的性能和上下文长度，又将自身对Worker模型的负面影响控制在很低的水平（<10%）。
    -   **与现有技术兼容**：设计与连续批处理、分块预填充、PagedAttention等现有优化技术正交，易于集成。

-   **不足与局限**：
    -   **硬件依赖性强**：核心设计依赖于同处一个服务器并通过NVLink直接连接的GPU。扩展到多服务器环境会面临巨大的通信开销和硬件挑战，这也在论文中作为未来工作指出。因此，其应用场景局限于单服务器、多GPU的部署。
    -   **实验场景可能不全面**：实验主要基于模拟的工作负载（Poisson到达过程），未在真实的生产流量（如突发高峰、长时间空闲）下测试。此外，图9中对最大上下文长度的测试声明“不考虑模型配置指定的最大长度”，虽然能突出方法优势，但可能不完全符合实际生产环境的部署约束。
    -   **潜在的倾斜风险**：Worker模型的干扰比（≤9.7%）在文中被评价为“极小”，但在对延迟极度敏感的线上服务中，这个程度的性能下降可能仍不可接受。论文未详细讨论Worker模型在极端负载（如自身请求量接近饱和）下的表现。
    -   **适用范围局限**：系统结构（一个Master带动多个Workers）天然倾向于资源不对等。如果Workers本身也承担高KV缓存需求的任务（如长上下文），则其共享资源的能力会迅速枯竭，系统优势将不再明显。论文未探索Master和Worker角色动态切换或更复杂的多Master-多Worker协作模式。
