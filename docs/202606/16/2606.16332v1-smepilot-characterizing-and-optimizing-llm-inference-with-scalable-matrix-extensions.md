## 摘要
现代CPU越来越多地集成矩阵扩展（例如Arm可扩展矩阵扩展SME），可在CPU内提供高吞吐量的矩阵执行。然而，对于大语言模型推理而言，这些单元并不能普遍替代传统CPU核心：预填充、解码、注意力机制和KV缓存操作表现出不同的算术强度、向量行为和布局要求，同时SME单元与CPU核心仍需竞争共享内存带宽。本文通过基于屋顶线模型的SME使能CPU表征来研究这种不匹配，并利用所得模型指导算子级执行选择。我们提出SMEPilot，一种大语言模型推理引擎，能够针对每个算子形状选择仅CPU、仅SME或SME+CPU协同执行。SMEPilot以图块粒度将矩阵工作划分到SME和CPU核心上，在注意力机制中将适合SME的矩阵阶段与适合CPU的向量阶段重叠执行，并维持布局状态，使得打包的张量表示得以复用，而非在关键路径上重复构建。在手机、PC和服务端平台上，针对Llama-3.2-3B、Qwen3-4B和Qwen3-30BA3B模型，SMEPilot将端到端推理性能提升了最高3.94倍。

## Abstract
Modern CPUs increasingly integrate matrix extensions, such as Arm Scalable Matrix Extension (SME), that provide high-throughput matrix execution within the CPU. For LLM inference, however, these units are not a universal replacement for conventional CPU cores: prefill, decode, attention, and KV-cache operations expose different arithmetic intensities, vector behavior, and layout requirements, while SME units and CPU cores still compete for shared memory bandwidth. This paper studies this mismatch through a roofline-based characterization of SME-enabled CPUs and uses the resulting model to guide operator-level execution choices. We present SMEPilot, an LLM inference engine that selects CPU-only, SME-only, or cooperative SME+CPU execution for each operator shape. SMEPilot partitions matrix work across SME and CPU cores at tile granularity, overlaps SME-suitable matrix stages with CPU-suitable vector stages in attention, and maintains layout state so packed tensor representations are reused rather than repeatedly rebuilt on critical paths. Across Llama-3.2-3B, Qwen3-4B, and Qwen3-30BA3B on phone, PC, and server platforms, SMEPilot improves end-to-end inference performance by up to 3.94$\times$.

# 论文总结：SMEPilot: Characterizing and Optimizing LLM Inference with Scalable Matrix Extensions

## 1. 核心问题与整体含义（研究动机和背景）

现代 CPU 日益集成矩阵扩展单元（如 Arm 的 Scalable Matrix Extension, SME），可在 CPU 内部提供高吞吐的矩阵指令。然而，对于大语言模型推理，SME 并非能简单替代传统 CPU 向量核心。原因在于：
- LLM 推理的不同算子（prefill、decode、attention、KV-cache）具有差异极大的算术强度（低到高）和向量行为。
- SME 单元与 CPU 核心共享内存带宽，但 SME 只提高计算峰值，不提高带宽上限。
- 直接将 SME 作为 drop-in 替换会导致部分场景下硬件利用率不足（大矩阵时一方闲置）或性能退步（小矩阵时 SME 延迟高）。
因此，需要系统研究如何协同使用 SME 和 CPU 核心，以最大化整体推理性能。

## 2. 论文提出的方法论：核心思想、关键技术细节

**核心思想**：基于多天花板屋顶线模型（Multi-Ceiling Roofline Model）在线选择每个算子的执行模式（CPU-only、SME-only 或混合 SME+CPU），并通过图块级分区、阶段流水线和布局感知运行时消除三种执行间隙。

**关键技术细节**：

1. **多天花板屋顶线模型**  
   - 针对每个算子计算算术强度 \(I_o = W_o/Q_o\)。
   - 共享内存带宽 \(B\)，不同执行目标（CPU、SME、混合）有各自的计算峰值 \(P_h\)。
   - 根据算术强度与各目标脊点 \(P_h/B\) 的关系，将算子分为：  
     - **内存受限区**（低于 CPU 脊点）→ CPU-only  
     - **脊线穿越区**（CPU 与 SME 脊点之间）→ SME-only  
     - **计算受限区**（高于 SME 脊点）→ 混合 SME+CPU  
   - 非矩阵操作（softmax、激活等）始终放 CPU。

2. **图块级工作分区（Tile-level Work Partitioning）**  
   - 对 GEMM 算子沿输出维度（\(M\) 或 \(N\)）分块，不同块分给 SME 和 CPU 工人，避免跨工人归约。
   - 分区比率由在线规划器根据 Profiled 的有效吞吐和内核块对齐决定，使两端同时完成。
   - 对 MoE 专家，按路由 token 数分类：小 token 专家用 CPU，大 token 专家可混合执行。

3. **阶段感知流水执行（Phase-aware Pipeline Execution）**  
   - 注意力操作分解为矩阵阶段（\(QK^T\), \(PV\)）和向量阶段（softmax）。  
   - 允许不同图块同时处于不同阶段：SME 处理矩阵阶段的后续图块，CPU 处理向量阶段的前序图块。
   - 图块大小由离线配置和在线规划平衡；支持掩码感知的矩阵内核（跳过全掩蔽块）。

4. **布局感知运行时（Layout-aware Runtime）**  
   - 跟踪张量布局作为运行时状态，转换仅在消费者需要不同布局时插入。
   - 静态权重：在模型加载时一次性打包为 SME 友好布局，脱离请求关键路径。
   - 在线激活（如 KV-cache）：在生产侧（K/V 投影输出时）直接写入 SME 打包布局，避免后续注意力调用时重复打包。

**规划流程**：  
- 离线 Profiler 测量 SME 和 CPU 的计算吞吐、内存带宽等硬件指标。  
- 在线规划器结合请求形状和模型计算图，对每个算子输出执行计划（执行目标、线程分配、分区形状）。
