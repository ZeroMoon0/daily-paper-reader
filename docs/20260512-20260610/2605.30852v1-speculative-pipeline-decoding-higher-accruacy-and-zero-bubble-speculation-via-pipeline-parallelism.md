## 摘要
投机解码（Speculative Decoding, SD）通过采用草拟-验证范式加速低并发大语言模型（LLM）推理。然而，主流方法通常依赖于多令牌预测，这引入了不断增加的预测难度和串行草拟延迟。为了解决这些问题，我们提出了投机流水线解码（Speculative Pipeline Decoding, SPD），这是一个开创性的框架，释放了流水线并行的真正潜力。通过将目标LLM划分为 $n$ 个流水线阶段，SPD 允许 LLM 并行处理 $n$ 个令牌以加速解码。为了在单序列解码中持续填充流水线，一个投机模块聚合不同流水线深度的中间特征来预测下一个令牌，该模块与目标模型的流水线步骤严格并行执行，从而实现有界的难度、更高的接受率和零延迟气泡。我们的实验表明，与主流基线相比，SPD 实现了显著更高的理论加速比，为 LLM 解码加速提供了一种高度可扩展的解决方案。我们的代码可在 https://github.com/yuyijiong/speculative_pipeline_decoding 获取。

## Abstract
Speculative Decoding (SD) accelerates low-concurrency LLM inference by employing a draft-then-verify paradigm. However, mainstream methods typically rely on multi-token prediction, which introduces escalating prediction difficulty and serial drafting latency. To address these, we propose Speculative Pipeline Decoding (SPD), a groundbreaking framework that unlocks the true potential of pipeline parallelism. By partitioning the target LLM into $n$ pipeline stages, SPD allows LLM to process $n$ tokens in parallel to accelerate decoding. To continuous fill the pipeline in single sequence decoding, a speculation module aggregates intermediate features across different pipeline depths to predict the next token, executing strictly in parallel with the target model's pipeline step, to realize bounded difficulty, higher acceptance rates, and zero latency bubbles. Our experiments demonstrate that SPD achieves a significantly higher theoretical speedup compared to mainstream baselines, offering a highly scalable solution for LLM decoding acceleration. Our code is available at https://github.com/yuyijiong/speculative_pipeline_decoding

# 论文详细总结

## 1. 论文的核心问题与整体含义（研究动机和背景）

大型语言模型（LLM）的自回归解码受限于内存带宽瓶颈。投机解码（Speculative Decoding, SD）通过“草稿-验证”范式加速推理，但主流方法（如 EAGLE 系列）均依赖多 token 预测：草稿模型需依次预测多个未来 token，这导致两个根本性问题：
- **复合预测难度**（长距离衰减）：随预测步数增加，草稿模型的浅层特征与目标模型的真实分布逐渐偏离，接受率急剧下降。
- **串行延迟开销**：草稿生成过程迫使目标模型闲置，产生等待气泡，且草稿本身的串行操作也占据额外时间。

现有改进（如 P-EAGLE、Speculative Speculative Decoding）难以根本解决或引入了新的权衡（如计算量剧增）。为此，本文提出 **Speculative Pipeline Decoding (SPD)**，通过将 LLM 划分为 $n$ 个流水线阶段，并行处理多个 token，彻底消除气泡，并利用多深度特征聚合实现可接受的预测准确率。

## 2. 论文提出的方法论

- **核心思想**：将目标 LLM 的所有层均匀划分为 $n$ 个流水线阶段；每步流水线前向步同时处理 $n$ 个 token（各处于不同深度）。为持续向流水线提供新 token，引入一个**流水线投机模块**，在目标模型执行流水线步骤的同时，预测下一个 token。
- **关键技术细节**：
  - **多深度特征聚合**：根据 token 当前完成的阶段数 $k$，提取其三个层级的隐藏状态（嵌入层、中间层、最深层），拼接后经全连接层投影得到特征 $g_t^k$：
    $$g_t^k = \text{FC}\left(\text{Concat}\left( H_t^0,\; H_t^{l_{\max}/2},\; H_t^{l_{\max}} \right)\right)$$
  - **输入状态调度**：投机模块在流水线步骤**开始前**读取各 token 的当前特征（即输入状态），因此与目标模型完全并行执行，无等待气泡。特征序列 $\mathcal{G}_t$ 按稳态公式（2）构建：
    $$\mathcal{G}_t = \left[ g_1^n, g_2^n, \dots, g_{t-n}^n,\; g_{t-n+1}^{n-1},\; \dots,\; g_{t-1}^1,\; g_t^0 \right]$$
  - **投机模块**：由一个单层或多层 Transformer Decoder 和 LM Head 构成，根据 $\mathcal{G}_t$ 预测 $x_{t+1}$。若模块层数 $L_s \leq L/n$，其延迟可被目标流水线步骤完全隐藏。
  - **验证与回滚**：每步流水线结束后，最旧 token 完成最终阶段，其 logits 用于验证流水线中第二旧的 token（此前已被投机）。若拒绝，触发 KV 缓存截断、流水线冲刷和重新播种。
  - **训练**：冻结目标 LLM，仅更新投机模块。使用知识蒸馏（KL 散度）让模块学习逼近目标模型的 logits 分布。训练时通过模拟流水线填充（随机决定活跃尾部位数）使输入分布与推理一致。

## 6. 论文的主要结论与发现

- SPD 在大多数配置下取得**最高的理论加速比**，特别是高温度（$T=1$）时优势明显。例如 Qwen3.5-9B 上 HumanEval 任务 $n=16, L_s=2, W=4, T=1$ 时达到 $5.97$，远超 EAGLE-3 的 $3.12$。
- **可扩展性优异**：增加 $n$ 或 $L_s$ 均能稳定提升加速比，不受复合难度限制；而 EAGLE-3 增大 $m$ 时加速比反而下降（因草稿开销增大），PPSD 增大 $n$ 时接受率停滞。
- **投机延迟完全隐藏**：通过输入状态调度实现，消融实验证明若改为输出状态，原始接受率虽升高，但引入等待气泡后理论加速比反而大幅下降（如 $n=16$ 时从 $>3$ 降至 $1.83$）。
- **高温度鲁棒性**：SPD 在随机采样下减速幅度远小于 EAGLE-3，因其特征更丰富，能更好捕捉完整 logit 分布。
- **草稿树有效**：$W=4$ 相比 $W=1$ 进一步提升了 SPD 的加速比，且在高温度下增益更显著。

## 7. 优点

- **零气泡架构**：投机模块与目标模型完全并行，消除了传统 SD 中的串行等待，这使得 SPD 的理论加速比完全由接受率决定，不受草稿延迟拖累。
- **预测难度有界**：多深度特征聚合使得每个 token 的特征缺失深度最多为 $n$ 层，避免传统方法中长距离分布的指数级恶化。
- **高可扩展性**：增加流水线阶段数 $n$ 或投机模块深度 $L_s$ 不会引入额外延迟，直接转化为加速比提升。
- **训练简单高效**：仅需 1 epoch 知识蒸馏，训练数据规模适中，参数冻结确保输出分布不变。
- **支持草稿树**：可自然扩展为树结构推测，进一步利用 beam 搜索提高接受率。

## 8. 不足与局限

- **缺乏端到端工程优化**：当前实现基于 PyTorch，缺少异步执行、自定义 CUDA 核、连续批处理等系统级优化，因此无法测量实际墙钟加速比。论文承认这是工程挑战而非算法缺陷。
- **异构架构负载不平衡**：对于具有混合注意力层（如 Qwen3.5-4B 中标准注意力和线性注意力交替）的模型，划分成 $n$ 阶段后各阶段计算负载可能不均，破坏同步，产生气泡。论文指出可通过选择同构架构或更多层模型来缓解。
- **训练中的近似注意力**：为降低扩展序列的注意力矩阵大小，仅将 $g^0$ 块作为查询，而其他深度块只提供键值。这种非对称设计可能降低预测精度（论文承认是折中）。
- **分布式通信开销未考虑**：流水线并行在跨 GPU 执行时会引入通信延迟，论文理论分析假设单步流水线代价为 $1/n$，实际多 GPU 场景下通信可能成为瓶颈。
- **对比设置不完全对等**：SPD 允许 $L_s>1$ 而基线 EAGLE-3 固定 $L_s=1$，虽然论文给出了理由，但若 EAGLE-3 也增加 $L_s$ 其理论加速比公式会相应变化，可能降低对比的绝对公平性。
